     9→### Shared types (`src/lib/types.ts`)
    10→```ts
    11→export interface Song {
    12→  id: string;
    13→  title: string;
    14→  prompt: string;
    15→  lyrics: string;
    16→  genre: string;
    17→  mood: string;
    18→  style: string;
    19→  voice: string;
    20→  audioUrl: string;        // "/api/audio/{id}"
    21→  audioFormat: string;     // "mp3"
    22→  durationMs: number;
    23→  createdAt: string;       // ISO string
    24→}
    25→```
    26→
    27→### API contract
    28→- `POST /api/generate`
    29→  - body: `{ prompt: string; genre: string; mood: string; style: string; voice?: string }`
    30→  - 200: `Song` (without audio bytes; includes `audioUrl`)
    31→  - 400: `{ error: string }` ; 500: `{ error: string }`
    32→- `GET /api/songs` → 200: `{ songs: Song[] }` (newest first)
    33→- `DELETE /api/songs/[id]` → 200: `{ success: true }` ; 404: `{ error: string }`
    34→- `GET /api/audio/[id]` → 200 `audio/mpeg` stream ; 404 if missing
    35→
    36→### AI service layer (Agent 3 owns `src/lib/ai/`)
    37→- `src/lib/ai/lyrics-generator.ts`
    38→  - `generateLyrics({ prompt, genre, mood, style }): Promise<{ title: string; lyrics: string }>`
    39→  - Uses LLM; returns title + lyrics (lyrics capped ~900 chars to fit TTS budget).
    40→- `src/lib/ai/audio-synth.ts` (the "Ace Music" adapter)
    41→  - `synthesizeAudio({ text, voice?, speed? }): Promise<{ buffer: Buffer; format: 'mp3' }>`
    42→  - Uses TTS; chunks text >1024 chars, generates mp3 per chunk, concatenates buffers.
    43→- `src/lib/ai/index.ts` re-exports both.

    95→- ⚠️ IMPORTANT FINDING for the orchestrator / Agent 2 — TTS FORMAT DEVIATION FROM SPEC:
    96→  - The SDK skill doc claimed `response_format: 'mp3'` is supported in non-streaming mode. The LIVE TTS SERVER REJECTS `'mp3'` with HTTP 400 `{"error":{"code":"1214","message":"不支持当前response_format值"}}` ("unsupported response_format value"). Verified empirically with `z-ai-web-dev-sdk@0.0.18`.
    97→  - Server DOES accept: `response_format: 'wav'` (→ `audio/wav`, RIFF/WAVE), `'pcm'` (→ raw samples), or omitted (→ `audio/pcm`).
    98→  - Therefore `audio-synth.ts` requests `response_format: 'wav'` and `SynthResult.format` is typed as `'wav' | 'mp3'` (loosened from the spec's literal `'mp3'`; runtime value is `'wav'`). This is strictly more permissive, so any caller code that does `result.format === 'mp3'` still type-checks.
    99→  - The multi-chunk merger (`mergeWavBuffers`) parses each RIFF/WAVE chunk, concatenates the PCM payloads, and rebuilds one canonical 44-byte header — so multi-chunk playback is seamless with correct duration (naive `Buffer.concat` would leave mid-stream RIFF headers some browsers refuse).
   100→  - ACTION NEEDED by Agent 2 / orchestrator: the API contract in worklog.md says `GET /api/audio/[id]` returns `audio/mpeg`. Because the adapter now returns WAV, the `/api/audio/[id]` route must set `Content-Type: audio/wav` (recommended: derive it from the stored `audioFormat` field rather than hardcoding). The `Song.audioFormat` field in `types.ts` is already typed as a plain `string` (not literal `'mp3'`), so it can hold `'wav'` with no type change. The `generate` route should persist `audioFormat: 'wav'` from `SynthResult.format`. Recommend the orchestrator update the contract line in worklog.md from `audio/mpeg` to `audio/wav` (or to "derived from audioFormat").
   101→- Parallel-processing requirement satisfied: all TTS chunk calls run concurrently via `Promise.all`.
   102→- Swappable-adapter contract preserved: only `audio-synth.ts` needs to change when the real Ace Music model lands; the exported signature stays the same (callers consume `format` dynamically).

  1204→
  1205→#### Profile page — Prisma direct vs. API fetch
  1206→
  1207→The spec allows either ("Fetches the profile data server-side from the API (or
  1208→directly via Prisma `db`)"). Chose **Prisma direct** with a minimal `select:
  1209→{ id: true }` because:
  1210→
  1211→- `<ProfileView>` is a client component that fetches its own data via
  1212→  `GET /api/profile/[username]` (with full loading skeleton + 404 handling
  1213→  baked in). The server component only needs to (a) verify existence and
  1214→  (b) grab the owner's `id` for the `isOwnProfile` comparison. Pulling the

  1217→  roundtrip back to the same Next.js server.
  1218→- Username is normalized (trim + lowercase) before lookup — matches the API
  1219→  route's normalization so `/u/JohnDoe` and `/u/johndoe` resolve to the same
  1220→  user, and the lowercase lookup hits the unique index.
  1221→
  1222→#### `isOwnProfile` computation
  1223→
  1224→```ts
  1225→const session = await getServerSession(authOptions);
  1226→const sessionUserId = (session?.user as { id?: string } | undefined)?.id;
  1227→const isOwnProfile = !!sessionUserId && sessionUserId === profileUser.id;
  1228→```
  1229→
  1230→- Uses `getServerSession(authOptions)` exactly as the spec requires (not the
  1231→  `getCurrentUserId()` helper from `src/lib/session.ts` — the spec named the
  1232→  primitives).
  1233→- The cast `(session.user as { id?: string }).id` matches the pattern in
  1234→  `src/lib/session.ts` — the `id` field is embedded in the JWT by the `jwt`
  1235→  callback in `auth.ts` and exposed via the `session` callback, but isn't on

  1237→  pattern.

  1242→
