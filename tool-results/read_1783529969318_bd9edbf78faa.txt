    27‚Üí### API contract
    28‚Üí- `POST /api/generate`
    29‚Üí  - body: `{ prompt: string; genre: string; mood: string; style: string; voice?: string }`
    30‚Üí  - 200: `Song` (without audio bytes; includes `audioUrl`)
    31‚Üí  - 400: `{ error: string }` ; 500: `{ error: string }`
    32‚Üí- `GET /api/songs` ‚Üí 200: `{ songs: Song[] }` (newest first)
    33‚Üí- `DELETE /api/songs/[id]` ‚Üí 200: `{ success: true }` ; 404: `{ error: string }`
    34‚Üí- `GET /api/audio/[id]` ‚Üí 200 `audio/mpeg` stream ; 404 if missing
    35‚Üí
    36‚Üí### AI service layer (Agent 3 owns `src/lib/ai/`)
    37‚Üí- `src/lib/ai/lyrics-generator.ts`
    38‚Üí  - `generateLyrics({ prompt, genre, mood, style }): Promise<{ title: string; lyrics: string }>`
    39‚Üí  - Uses LLM; returns title + lyrics (lyrics capped ~900 chars to fit TTS budget).
    40‚Üí- `src/lib/ai/audio-synth.ts` (the "Ace Music" adapter)
    41‚Üí  - `synthesizeAudio({ text, voice?, speed? }): Promise<{ buffer: Buffer; format: 'mp3' }>`
    42‚Üí  - Uses TTS; chunks text >1024 chars, generates mp3 per chunk, concatenates buffers.
    43‚Üí- `src/lib/ai/index.ts` re-exports both.

    95‚Üí- ‚öÝÔ∏è IMPORTANT FINDING for the orchestrator / Agent 2 ‚Äî TTS FORMAT DEVIATION FROM SPEC:
    96‚Üí  - The SDK skill doc claimed `response_format: 'mp3'` is supported in non-streaming mode. The LIVE TTS SERVER REJECTS `'mp3'` with HTTP 400 `{"error":{"code":"1214","message":"‰∏çÊîØÊåÅÂΩìÂâçresponse_formatÂÄº"}}` ("unsupported response_format value"). Verified empirically with `z-ai-web-dev-sdk@0.0.18`.
    97‚Üí  - Server DOES accept: `response_format: 'wav'` (‚Üí `audio/wav`, RIFF/WAVE), `'pcm'` (‚Üí raw samples), or omitted (‚Üí `audio/pcm`).
    98‚Üí  - Therefore `audio-synth.ts` requests `response_format: 'wav'` and `SynthResult.format` is typed as `'wav' | 'mp3'` (loosened from the spec's literal `'mp3'`; runtime value is `'wav'`). This is strictly more permissive, so any caller code that does `result.format === 'mp3'` still type-checks.
    99‚Üí  - The multi-chunk merger (`mergeWavBuffers`) parses each RIFF/WAVE chunk, concatenates the PCM payloads, and rebuilds one canonical 44-byte header ‚Äî so multi-chunk playback is seamless with correct duration (naive `Buffer.concat` would leave mid-stream RIFF headers some browsers refuse).
   100‚Üí  - ACTION NEEDED by Agent 2 / orchestrator: the API contract in worklog.md says `GET /api/audio/[id]` returns `audio/mpeg`. Because the adapter now returns WAV, the `/api/audio/[id]` route must set `Content-Type: audio/wav` (recommended: derive it from the stored `audioFormat` field rather than hardcoding). The `Song.audioFormat` field in `types.ts` is already typed as a plain `string` (not literal `'mp3'`), so it can hold `'wav'` with no type change. The `generate` route should persist `audioFormat: 'wav'` from `SynthResult.format`. Recommend the orchestrator update the contract line in worklog.md from `audio/mpeg` to `audio/wav` (or to "derived from audioFormat").
